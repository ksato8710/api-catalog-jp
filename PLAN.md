### 1-2. ビジョン & ミッション

- **ビジョン**: 日本のすべての開発者が、言語の壁なく世界中の API を発見・理解・活用できる世界をつくる
- **ミッション**: API の民主化を通じて、個人開発者やスタートアップのプロダクト開発を加速する
- **トーン**: 実用的で親しみやすい技術ガイド。難解な専門用語を避け、隣で一緒に調べる同僚のようなスタンス

### 1-3. ターゲットペルソナ

### 1-4. 市場環境 — なぜ今やるのか

#### 市場の構造的変化（2022〜2026年）

| 出来事 | 影響 |
|---|---|
| **ProgrammableWeb 終了**（2022年10月） | 17年の歴史、24,000件の API 情報が消失。中立的ディレクトリの後継が不在 |
| **RapidAPI の縮小**（Nokia 買収 2024年11月） | アクティブユーザー「数百万→数千」に激減。5G/ネットワーク API に戦略転換 |
| **Postman の台頭** | API Network（12万+ コレクション）が最大規模に成長。2026年3月に MCP サーバーディレクトリ追加予定 |
| **GitHub public-apis** | 39.8万 Star のデファクト API 発見起点。ただし Markdown リストで検索・フィルタ機能なし |
| **AI 時代の変化** | Postman の MCP 対応、Apidog の AI 意味検索など、AI 対応が差別化要因に |

#### 市場の空白

```
日本語の本格的 API カタログの数: 0
日本のインターネットユーザー: 9,500万+
日本のソフトウェア開発者: 180万+（推定）
```

**結論: 日本語 API カタログは完全なブルーオーシャン。**

#### 最大の脅威: Apidog

- 日本語 UI を持つ唯一のグローバルプレイヤー。AI 意味検索、テスト環境、ドキュメント自動生成を統合
- **弱み**: 日本特化ではない（PayPay/LINE 等の国内 API は未収録）、人気根拠の表示なし
- **対策**: 日本ローカル API × 日本語コンテンツの深さで差別化。ツール統合では戦わない

### 1-7. ポジショニング — 勝てるポイントと戦わない領域

**勝てるポイント（差別化要因）**:
1. **唯一の完全日本語カタログ** — グローバル競合で日本語 UI を持つのは Apidog のみ。日本特化ではない
2. **人気スコアの根拠表示** — 利用企業名、参照元 URL、市場データを明示。他サイトにはない透明性
3. **日本ローカル API の収録** — PayPay, LINE, 楽天, 気象庁, e-Stat 等。グローバルサイトでは見つからない
4. **統一フォーマット** — 認証・料金・レート制限・SDK を全 API 横断で比較可能

**戦わない領域**:
- ブラウザ上の API テスト機能（Postman/Apidog の土俵）
- マーケットプレイス（API の仲介・課金）
- 英語圏のユーザー獲得

## Part 4: 意思決定ガイドライン

AIエージェントが作業中に判断に迷った場合、以下の原則に従うこと。

### 戦略的優先順位（上が最優先）

1. **日本語の質と正確性** — 機械翻訳レベルの日本語は NG。自然で実用的な日本語であること
2. **日本ローカル API の充実** — グローバル API より日本ローカル API を優先追加（差別化の源泉）
3. **SEO 効果** — 全コンテンツは「○○ API 日本語」「○○ API 比較」で検索される前提で書く
4. **データの透明性** — 人気スコアには必ず根拠（利用企業、参照元 URL）を付ける
5. **運用の自動化** — 手作業を増やす施策より、自動化できる施策を優先
6. **ユーザー体験** — 機能追加はシンプルに。過剰なインタラクションは避ける

### コンテンツ品質基準

- API 説明文は **3〜5行の日本語** で、用途がすぐわかる具体性
- 人気スコアの `sources` には **最低2件の参照元 URL** を含める
- 料金情報は **無料枠の有無と概算コスト** を必ず含める
- `difficulty` は **初心者が使えるか** を基準に判定（ドキュメントの充実度、SDK 有無）
- `region` は **日本から利用可能か、日本語ドキュメントがあるか** で判定

### 戦わない判断

以下の方向に進む提案が出た場合は却下する:
- ブラウザ上の API テスト機能の実装（Postman の土俵）
- 英語版の作成（リソースの分散）
- 有料課金の仲介（マーケットプレイスモデル）
- ネイティブアプリ化（静的サイトの低コスト運用を維持）
- ユーザー認証が必要な機能の導入（Phase 3 より前は不要）
