1. 市場調査サマリー

2022年にProgrammableWeb（API数24,000件）が閉鎖し、中立的なAPIディレクトリ市場に大きな空白が生まれた。 RapidAPIはNokia買収後に縮小傾向、PostmanはAPI Network拡充を進めるが日本語非対応。 日本語に特化したAPIカタログは事実上存在せず、明確なブルーオーシャンが確認された。

調査で判明した主要トレンド
  • ProgrammableWeb終了の影響: 2022年10月に17年の歴史に幕。Salesforce/MuleSoftが運営終了。24,000件のAPI情報が消失し、中立的ディレクトリの後継が不在
  • RapidAPIの縮小: Nokia買収（2024年11月）後、アクティブユーザーが「数百万→数千」に激減。5G/ネットワークAPIに戦略転換し、パブリックマーケットプレイスの優先度低下
  • Postmanの台頭: API Network（12万+コレクション）が最大規模に成長。2026年3月にAPI Catalog機能とMCPサーバーディレクトリを追加予定
  • GitHubの存在感: public-apis（GitHub 39.8万Star）がデファクトの無料API発見起点に。ただしMarkdownリストで検索・フィルタ機能なし
  • AI時代の変化: PostmanのMCPサーバーディレクトリ、Apidogの意味検索（Algolia AI）など、AI対応が差別化要因に

グローバル市場の構造的変化

APIカタログ市場は「中立ディレクトリ」から「ツール統合型プラットフォーム」へシフトしている。 ProgrammableWeb型の「調べるだけ」のサイトは衰退し、Postman/Apidogのように「発見→テスト→実装」を 一気通貫で提供するサービスが優勢。ただし、いずれも英語圏が主戦場で、 日本語市場は戦略的優先度が低い。

最大の脅威: Apidog

Apidog API Hub

日本語UIを持つ唯一のグローバルプレイヤー。1,800+ APIを収録し、AI意味検索、 テスト環境、ドキュメント自動生成を統合。2025年末にAlgolia AIによる セマンティック検索を導入し、急成長中。

  • 強み: 資金力、開発ツール統合、多言語対応
  • 弱み: 日本特化ではない（PayPay/LINE等の国内APIは未収録）、人気根拠の表示なし
  • 対策: 日本ローカルAPI×日本語コンテンツの深さで差別化
間接競合: Qiita / Zenn

技術記事プラットフォーム

「○○ API 使い方」の検索結果でQiita/Zenn記事が上位を占める。 APIの「発見」ではなく「使い方」で検索する日本の開発者にとって、 事実上の情報源。

  • 強み: SEOの強さ、コミュニティの信頼、個人の実体験
  • 弱み: 情報の断片化、フォーマット不統一、陳腐化
  • 対策: カタログからの送客先として共存。API記事のハブとして機能

勝てるポイント（差別化要因）
  • 唯一の完全日本語カタログ: グローバル競合で日本語UIを持つのはApidogのみ。日本特化ではない
  • 人気スコアの根拠表示: 利用企業名、参照元URL、市場データを明示。他サイトにはない透明性
  • 日本ローカルAPIの収録: PayPay, LINE, 楽天, 気象庁, e-Stat等。グローバルサイトでは見つからない
  • 統一フォーマット: 認証・料金・レート制限・SDKを全API横断で比較可能
負けているポイント（課題）
  • API数 100件（Phase 1完了）: Postman 12万+ / RapidAPI 4万+ / Public APIs 1,400+に対しまだ少ない。Phase 2で300件を目指す
  • テスト機能なし: Postman/RapidAPI/Apidogはブラウザ上でAPI実行可能
  • コミュニティ不在: ユーザー投稿・レビュー・コントリビュート機能がない
  • 自動更新なし: APIs.guruは週次自動更新。手動運用はスケールしない
  • SEO/認知度: 新規サイトのため検索流入・被リンクが皆無
5. SWOT分析
S - Strengths（強み）
  • 日本語での完全な情報提供（唯一のポジション）
  • 人気スコアの根拠と利用企業の可視化
  • 日本ローカルAPI（LINE, PayPay, 楽天等）の収録
  • モダンなUI/UXデザイン
  • 静的サイトによる低運用コスト
W - Weaknesses（弱み）
  • API掲載数が100件（Phase 1完了、Phase 2で300件目標）
  • テスト機能・インタラクティブ機能の欠如
  • 手動更新のみ（スケーラビリティ課題）
  • コミュニティ/ユーザー参加の仕組みなし
  • 人気スコアが自前測定ではなく二次情報ベース
O - Opportunities（機会）
  • 日本語APIカタログ市場は完全な空白地帯
  • ProgrammableWeb終了後の中立ディレクトリ需要
  • RapidAPI縮小による代替先の需要
  • 「○○ API 日本語」のSEOロングテール
  • AI/LLMの普及によるAPI利用拡大
  • Qiita/Zenn記事との相互送客
T - Threats（脅威）
  • Apidogの日本語対応拡充
  • Postmanの多言語展開（未定だが可能性あり）
  • LLM（ChatGPT等）がAPI推薦を直接行う可能性
  • API情報の陳腐化リスク
  • 個人運営のリソース制約
6. 成長ロードマップ

4フェーズで「日本語APIカタログのデファクト」を目指す。最重要課題はAPI数の拡充とSEO資産の構築。

基盤強化 -- 100件達成 & SEO基盤構築

API掲載数を100件に拡充。カテゴリ別比較表を追加。独自ドメイン取得とSEO最適化。 「○○ API 日本語」のロングテールキーワードで検索上位を狙う。 各APIの個別ページ（/api/stripe など）を生成し、SEOの受け皿を作る。

リスクと対策
  • RISK コンテンツ制作のボトルネック（42→1,000件の拡充） → AI補助による半自動生成 + テンプレート化 + コミュニティ投稿
  • RISK API情報の陳腐化（料金改定、サービス終了等） → 定期クロール自動化 + ユーザー報告機能 + 更新日の明示
  • RISK Apidogの日本市場本格参入 → 日本ローカルAPI×比較コンテンツの深さで差別化。ツール統合では戦わない
  • RISK LLMがAPI推薦を直接行い、カタログサイト自体の需要消失 → MCP対応・構造化データ提供でLLMの参照元としてのポジションを確保
  • RISK SEO競争での敗北（Qiita/Zenn等の既存ドメインパワー） → 「○○ API 比較」「○○ API 日本語」等のニッチキーワードに集中
Next Action（今後の最優先アクション）

  • 3. 個別APIページの生成: /api/{id} 形式の詳細ページを自動生成し、SEOの受け皿を作る

  • 5. Zenn/Qiita記事の公開: 「2026年版 決済API比較」等のコンテンツマーケティング開始
